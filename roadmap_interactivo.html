🎙️ MoshWasi.AI

Ruta Completa: Moshi Full Duplex + Wasi

Esta aplicación detalla la receta técnica paso a paso para implementar un asistente de voz inmobiliario en tiempo real. Combina el modelo Moshi (full duplex) para la interacción hablada natural y la API de Wasi con búsqueda semántica para recuperar inmuebles precisos.

🎯

Objetivo Principal

Construir un sistema donde el usuario pueda hablar con naturalidad, interrumpir, y buscar propiedades en segundo plano, obteniendo respuestas basadas 100% en datos reales del inventario inmobiliario.

📚

Fuentes Clave

  • 🔗 Repo Moshi: Implementación del modelo dos streams.
  • 🔗 Moshi LoRA: Fine-tuning para entendimiento especializado.
  • 🔗 Wasi API: Manejo de tokens y esquemas de propiedades.
🎙️

¿Qué es el "Full Duplex"?

A diferencia de los asistentes tradicionales (hablas → esperas → responde), Moshi escucha y procesa audio al mismo tiempo. Puede interrumpir, confirmar mientras buscas, y mantener una latencia casi nula, emulando una conversación humana real.

Las 3 Piezas de la Arquitectura

El sistema se compone de tres motores principales que interactúan entre sí. Esta separación garantiza que la IA no "invente" (alucinaciones) propiedades, sino que funcione como una interfaz natural hacia una base de datos estricta.

Orden de Ejecución Recomendado

Para asegurar el éxito, el desarrollo debe seguir este orden estricto. Construir el motor de voz sin una capa de datos sólida resultará en alucinaciones.

1

Infraestructura & Data Layer (Etapas 1 y 2)

+

Objetivo: Preparar el terreno y los datos.

  • Configurar repositorios y entornos.
  • Crear el Wasi Inventory Service.
  • Normalizar los datos de Wasi (limpiar campos de texto libre, estandarizar monedas, geolocalización).
  • Implementar Sync Incremental (cronjob que actualiza tu DB local desde Wasi sin traer todo de cero).
2

Motor Semántico de Texto (Etapa 3)

+

Objetivo: Búsqueda en lenguaje natural antes de meter voz.

  • Implementar endpoint `/nlq-text` (Natural Language Query).
  • Procesar querys complejas: "Busco un depa de 2 cuartos en la zona norte, menos de 1000 usd".
  • Calcular afinidad y retornar JSON con las propiedades que hacen match y una justificación textual.
3

Moshi Core & WebSocket Gateway (Etapas 4 y 5)

+

Objetivo: Levantar la interacción conversacional duplex.

  • Desplegar Moshi en un entorno local estable (sin herramientas aún).
  • Construir el Voice Gateway: Servidor WebSocket que maneja el streaming de audio bidireccional.
  • Ajustar buffers y detectar VAD (Voice Activity Detection) para interrupciones limpias.
4

Tool Router & Cards UI (Etapa 6)

+

Objetivo: Conectar el cerebro con la voz.

  • Conectar Moshi con el endpoint Semántico (Tool Calling).
  • Moshi debe decir "Déjame buscarte opciones..." mientras el backend consulta a Wasi.
  • Enviar eventos al frontend para renderizar "Cards" visuales de los inmuebles encontrados mientras la voz los describe.
5

Integración, Prod & Fine-tuning (Etapas 8, 9, 10)

+

Objetivo: Pulir, asegurar y lanzar.

  • Integración Web/Mobile (cliente WebSocket).
  • Seguridad: Auth por JWT en el handshake del socket, Rate limiting.
  • Observabilidad (Métricas de latencia).
  • Opcional (Etapa 8): Fine-tuning LoRA de Moshi solo para mejorar modismos locales o el entendimiento del vocabulario inmobiliario (nunca para memorizar inventario).

Análisis: Desarrollo Desatendido (Vibe Coding)

En respuesta a tu consulta sobre plataformas low-code o IDEs para desarrollar esta arquitectura de forma casi desatendida (usando Claude, Antigravity, etc.). Debido a la complejidad de WebSockets, despliegue local de modelos IA (Moshi) y sincronización de bases de datos, las herramientas UI-only se quedan cortas. Se requiere un enfoque Agentic IDE.

Altamente Recomendado

💻 Cursor + Composer

Cursor con Claude 3.5 Sonnet es actualmente el estándar de oro. Su característica Composer (Ctrl+I) permite multi-file editing real.

  • ✓ Ventaja: Lee toda tu base de código (contexto global). Puedes decirle "Crea el WebSocket Gateway en Node y el cliente React".
  • ✓ Ventaja: Edita archivos directamente.
  • ⚠️ Desventaja: Aún requiere supervisión en arquitecturas complejas y manejo de errores de WebSockets.
Mayor Autonomía

🤖 Cline (VS Code Extension)

Anteriormente "Claude Dev". Es un agente autónomo que vive en tu IDE.

  • ✓ Ventaja: Verdadera ejecución desatendida. Puede ejecutar comandos en la terminal (ej. `npm install`, iniciar servidores), leer errores, iterar y corregir código sin tu intervención constante.
  • ✓ Ventaja: Ideal para desplegar Moshi localmente, ya que puede lidiar con problemas de dependencias de Python leyendo los logs.

🚀 Antigravity / Frameworks de Agentes

Plataformas orquestadoras con Claude integrado y "skills" (como consultar APIs, escribir código).

  • Veredicto: Si bien plataformas como Antigravity o frameworks como CrewAI/LangGraph son excelentes para crear pipelines de automatización, usarlos como reemplazo de un IDE para construir una app en tiempo real (WebSockets/Audio) suele ser frustrante. Son mejores para el backend/Data Layer (procesar los inmuebles de Wasi) que para compilar el gateway de voz.

⚡ Bolt.new / v0.dev

Herramientas de zero-setup (browser-based).

No recomendado para el backend Moshi. Excelentes para generar rápidamente el UI de las "Cards" y el chat frontend, pero la infraestructura de WebSockets, GPUs locales para Moshi y conexiones B2B a Wasi no pueden ejecutarse nativamente en estos entornos efímeros de navegador.

El Flujo de Trabajo Recomendado

Utiliza Cursor o VS Code + Cline. Inicia el prompt proporcionando exactamente el documento "Ruta completa.docx" al agente. Dile: "Actúa como Tech Lead. Vamos a ejecutar la Etapa 1 y 2. Escribe los scripts para conectarnos a Wasi API y normalizar los datos en SQLite/Postgres. Ejecuta las pruebas." Deja que el agente itere en la terminal hasta lograrlo, etapa por etapa. No intentes generar toda la app en un solo prompt.
